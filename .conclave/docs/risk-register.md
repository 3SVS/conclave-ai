# Risk Register

## R1. 너무 큰 범위를 한 번에 구현할 위험

- 위험: visual workspace, question engine, verification, autofix를 동시에 만들면 실패 가능성이 높다.
- 대응: Stage 0 탐색 후 Stage 1부터 하나씩 구현한다.

## R2. 일반 유저에게 개발자 용어가 노출될 위험

- 위험: PRD, requirement, acceptance, verification 같은 단어가 UI에 노출되면 어렵게 느껴진다.
- 대응: 사용자용 용어 매핑을 강제한다.

## R3. LLM 질문이 너무 많거나 부정확할 위험

- 위험: 사용자가 질문 폭탄을 맞거나, 쓸모없는 질문이 나온다.
- 대응: 질문 후보는 많이 만들되 처음에는 3~5개만 보여준다. 각 질문은 추천 답변과 이유를 포함한다.

## R4. 자동수정이 또 다른 버그를 만들 위험

- 위험: 실패 항목을 고치다 다른 기능을 깨뜨릴 수 있다.
- 대응: 자동수정은 사용자 승인 후 실행하고, 항목당 최대 2회만 시도한다. 수정 후 해당 항목을 다시 확인한다.

## R5. 확인 부족을 실패처럼 보이게 할 위험

- 위험: 사용자가 “제품이 틀렸다”고 느낄 수 있다.
- 대응: 확인 부족은 정상 상태로 설명한다. “확인할 근거가 부족해요”라고 표현한다.

## R6. 기존 Conclave 엔진과 중복 구현할 위험

- 위험: 이미 있는 review/autofix/audit/job 구조를 다시 만들 수 있다.
- 대응: Stage 0에서 기존 구조를 찾고 재사용 포인트를 문서화한다.

## R7. 비개발자와 개발자 모두에게 애매한 제품이 될 위험

- 위험: 너무 쉬운 UI는 개발자에게 부족하고, 너무 기술적인 UI는 일반 유저에게 어렵다.
- 대응: UI는 일반 유저용으로 만들고, 세부 evidence와 builder pack은 확장 패널 또는 다운로드로 제공한다.
