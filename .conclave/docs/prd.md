# 제품 설명서 — Conclave Visual Product Workspace

## 한 줄 설명

Conclave는 사용자의 러프한 아이디어를 제품 설명서와 만들 항목으로 정리하고, AI가 만든 결과물이 그 항목들을 실제로 만족하는지 확인 결과와 고쳐야 할 것을 시각적으로 보여주는 작업공간이다.

## 배경

현재 Conclave는 PRD 기반 PR 리뷰, multi-agent council, CLI, SaaS review/autofix 흐름을 갖고 있다. 하지만 CLI나 GitHub PR 중심 경험은 일반 유저에게 접근성이 낮다. 사용자는 터미널 명령어나 PR comment를 읽고 싶어 하지 않는다. 사용자는 자신의 아이디어가 제대로 제품으로 만들어지고 있는지 한 화면에서 보고 싶어 한다.

## 문제

일반 유저가 AI 개발 도구를 사용할 때 자주 겪는 문제는 다음과 같다.

1. 아이디어가 애매한 상태로 개발 AI에게 전달된다.
2. 개발 AI가 빈칸을 마음대로 추측한다.
3. 결과물이 그럴듯하지만 사용자가 원한 것과 다르다.
4. PR 리뷰는 코드 중심이라 빠진 요구사항을 놓친다.
5. 실패한 부분을 다시 어떤 식으로 고쳐야 할지 모른다.
6. CLI/PR 중심 도구는 비개발자에게 너무 어렵다.

## 목표

Conclave는 다음 경험을 제공해야 한다.

```text
아이디어 입력
→ Conclave가 이해한 내용 보여주기
→ 아이디어별 맞춤 질문
→ 제품 설명서 생성
→ 꼭 들어가야 할 항목 카드 생성
→ 결과물 연결
→ 항목별 확인 결과 표시
→ 안 맞음/확인 부족 항목 고쳐보기
→ 다시 확인
```

## 사용자용 핵심 화면

### 1. 아이디어

사용자가 만들고 싶은 제품을 자유롭게 입력한다.

예시:

```text
회의 녹음 파일을 올리면 자동으로 요약하고, 할 일을 뽑아서 Linear로 보내주는 앱을 만들고 싶어.
```

### 2. Conclave가 이해한 내용

Conclave가 사용자의 아이디어를 일반 언어로 요약한다.

예시:

```text
Conclave가 이해한 내용
- 사용자는 회의 녹음 파일을 올릴 수 있어야 함
- 녹음은 텍스트로 바뀌어야 함
- 회의 요약이 생성되어야 함
- 할 일이 따로 추출되어야 함
- 사용자가 선택한 할 일을 Linear로 보낼 수 있어야 함
```

### 3. 더 정확히 만들기 위한 질문

템플릿 질문만 던지지 않는다. 아이디어에 맞는 중요한 질문을 생성한다.

예시:

```text
Linear로 보내기 전에 사용자가 확인하게 할까요?
추천: 확인 후 보내기
이유: 잘못 추출된 할 일이 실제 업무 시스템에 들어가는 것을 막기 위해

[확인 후 보내기]
[자동으로 보내기]
[나중에 정하기]
[직접 입력]
```

### 4. 제품 설명서

사용자의 답변을 반영해 제품 설명서를 만든다. 이 문서는 사람이 읽을 수 있어야 하며, 내부적으로는 개발 AI에게 넘길 수 있는 구조화된 항목으로 변환 가능해야 한다.

### 5. 꼭 들어가야 할 항목

제품 설명서에서 만든 핵심 항목을 카드로 보여준다.

예시:

```text
REQ-001 사용자는 회의 녹음 파일을 올릴 수 있어야 함
REQ-002 잘못된 파일 형식을 올리면 이유를 알려줘야 함
REQ-003 회의 요약은 결정사항과 할 일을 분리해서 보여줘야 함
REQ-004 Linear로 보내기 전 사용자가 확인해야 함
REQ-005 사용자는 본인 회의록만 볼 수 있어야 함
```

사용자 UI에서는 `REQ-001` 같은 ID를 작게 보조 정보로만 보여주고, 핵심은 자연어 문장이다.

### 6. 확인 결과

결과물 또는 PR이 연결되면 각 항목별 상태를 보여준다.

```text
통과: 확인 근거가 있음
안 맞음: 확인 근거가 있는 실패
확인 부족: 검증할 증거가 부족함
결정 필요: 제품 결정을 더 해야 함
위험 감수: 사용자가 이번 버전에서 넘어가기로 함
```

### 7. 고쳐야 할 것

안 맞음 또는 확인 부족 항목을 모아 보여주고, 사용자가 다음 행동을 고를 수 있게 한다.

```text
안 맞는 항목
- Linear로 보내기 전 확인 단계가 없음
  [고쳐보기] [작업으로 만들기] [제품 설명서 수정] [위험 감수]

확인 부족
- 다른 사용자의 회의록 접근 차단 여부를 확인하지 못함
  [확인 방법 만들기] [개발자에게 확인 요청] [이번 버전에서 제외]
```

## 비목표

이번 방향의 초기 버전에서 하지 않는다.

- 완전 자동 개발 OS를 만들지 않는다.
- 모든 버그를 자동으로 찾는다고 약속하지 않는다.
- 모든 실패를 자동으로 고친다고 약속하지 않는다.
- 사용자에게 CLI 실행을 요구하지 않는다.
- GitHub PR comment만을 주요 UX로 삼지 않는다.
- 보안 제품처럼 모든 agent/MCP 위험을 보장한다고 표현하지 않는다.

## 성공 기준

초기 성공 기준은 다음과 같다.

1. 사용자가 raw idea를 입력할 수 있다.
2. Conclave가 이해한 내용을 자연어로 보여준다.
3. 아이디어에 맞는 맞춤형 질문 3~5개를 보여준다.
4. 사용자의 답변을 반영해 제품 설명서 초안을 만든다.
5. 제품 설명서에서 꼭 들어가야 할 항목 카드를 만든다.
6. 각 항목은 상태를 가질 수 있다.
7. 안 맞음/확인 부족 항목은 고쳐보기 흐름으로 들어갈 수 있다.
8. Claude Code 또는 Codex에 넘길 만들기 패키지를 생성할 수 있다.
9. 기존 Conclave review/autofix 엔진과 나중에 연결할 수 있는 데이터 구조가 있다.
