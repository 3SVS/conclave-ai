# UX Wireframe — 일반 유저용 Conclave 작업공간

## 핵심 원칙

- 사용자는 개발자가 아닐 수 있다.
- 어려운 영어와 개발자 용어를 최대한 숨긴다.
- 사용자는 “내가 생각한 제품이 제대로 만들어지고 있는지” 알고 싶다.
- 모든 화면은 다음 질문에 답해야 한다.

```text
내가 뭘 만들려고 했지?
Conclave는 어떻게 이해했지?
뭘 더 정해야 하지?
꼭 들어가야 하는 건 뭐지?
지금 어디까지 만들어졌지?
뭐가 안 맞지?
다음에 뭘 하면 되지?
```

## 메뉴 구조

```text
아이디어
제품 설명서
꼭 들어가야 할 것
만들기 진행상황
확인 결과
고쳐야 할 것
설정
```

## 화면 1 — 아이디어 입력

```text
무엇을 만들고 싶나요?

[ 큰 입력창 ]
예: 회의 녹음 파일을 올리면 자동으로 요약하고, 할 일을 뽑아서 Linear로 보내주는 앱

[빠르게 초안 만들기]
[정확하게 만들기]
[출시용으로 꼼꼼하게]
```

### 모드 설명

```text
빠르게 초안 만들기: 핵심 질문 3개만 묻고 시작
정확하게 만들기: 중요한 결정 8~12개를 정리
출시용으로 꼼꼼하게: 보안, 데이터, 예외상황까지 깊게 정리
```

## 화면 2 — Conclave가 이해한 내용

```text
Conclave가 이해한 내용

- 사용자는 회의 녹음 파일을 올릴 수 있어야 함
- 녹음은 자동으로 텍스트로 바뀌어야 함
- 회의 요약이 만들어져야 함
- 할 일이 따로 추출되어야 함
- 사용자가 고른 할 일을 Linear로 보낼 수 있어야 함

틀린 부분이 있나요?
[수정하기]
[맞아요, 계속]
```

## 화면 3 — 맞춤 질문

질문은 한 번에 너무 많이 보여주지 않는다. 3~5개씩 보여준다.

질문 카드 예시:

```text
결정 필요

Linear로 할 일을 보낼 때, 바로 보내도 될까요?

추천: 보내기 전에 사용자가 확인하게 하기
이유: 잘못 추출된 할 일이 실제 업무 시스템에 들어가는 걸 막기 위해

[사용자 확인 후 보내기]
[자동으로 보내기]
[나중에 정하기]
[직접 입력]
```

## 화면 4 — 제품 설명서

```text
제품 설명서

목표
회의 녹음을 올리면 요약과 할 일을 자동으로 정리하고, 사용자가 선택한 할 일을 Linear로 보낼 수 있게 한다.

이번 버전에 포함
- 녹음 파일 업로드
- 텍스트 변환
- 요약 생성
- 할 일 추출
- Linear 전송 전 확인

이번 버전에서 제외
- 팀 공유
- 결제
- 관리자 페이지
- 실시간 녹음
```

## 화면 5 — 꼭 들어가야 할 것

카드형 목록:

```text
꼭 들어가야 할 것 12개

[통과 전] 사용자는 회의 녹음 파일을 올릴 수 있어야 함
[통과 전] 잘못된 파일을 올리면 이유를 알려줘야 함
[통과 전] 요약은 결정사항과 할 일을 분리해서 보여줘야 함
[통과 전] Linear로 보내기 전 사용자가 확인해야 함
[통과 전] 사용자는 본인 회의록만 볼 수 있어야 함
```

## 화면 6 — 확인 결과

```text
전체 확인 결과

통과 5개
안 맞음 3개
확인 부족 2개
결정 필요 1개

안 맞음
1. Linear로 보내기 전 확인 단계가 없음
   확인 근거: 현재 구현은 할 일이 생성되면 바로 Linear API를 호출함
   [고쳐보기]

확인 부족
1. 다른 사용자의 회의록 접근 차단 여부를 확인하지 못함
   이유: 권한 테스트가 없고, DB 정책을 확인하지 못함
   [확인 방법 만들기]
```

## 화면 7 — 고쳐야 할 것

```text
Linear로 보내기 전 확인 단계가 없음

왜 안 맞나요?
제품 설명서에는 “사용자가 확인한 뒤 Linear로 보내야 한다”고 되어 있습니다.
하지만 현재 결과물은 할 일이 생성되면 바로 Linear로 보내는 흐름입니다.

고치는 방법
1. 전송 전 미리보기 화면 추가
2. 사용자가 보낼 항목을 선택
3. 사용자가 전송 버튼을 누른 뒤 Linear API 호출
4. 이 흐름을 확인하는 테스트 추가

[수정안 만들기]
[개발자에게 보낼 작업 만들기]
[제품 설명서 수정하기]
[이번 버전에서는 위험 감수]
```

## 화면 톤

- 비난하지 않는다.
- “실패했습니다”보다 “안 맞는 부분을 찾았어요”처럼 표현한다.
- “검증 불가”보다 “확인할 근거가 부족해요”처럼 표현한다.
- 사용자가 다음 버튼을 누르면 무엇이 일어나는지 명확해야 한다.
