# Product Decisions

## D1. 사용자 화면에서는 쉬운 한국어를 우선한다

내부적으로는 PRD, requirement, acceptance, verification 같은 단어를 써도 되지만, 일반 사용자 화면에서는 제품 설명서, 꼭 들어가야 할 항목, 확인 결과, 고쳐보기 같은 표현을 우선한다.

## D2. 질문은 맞춤형으로 생성한다

로그인 필요 여부, 결제 필요 여부 같은 공통 템플릿 질문만으로 끝내지 않는다. raw idea에서 핵심 행동, 외부 연동, 데이터, 권한, 실패 상황을 찾아 아이디어별 질문을 만든다.

## D3. 질문은 많이 만들되 처음에는 적게 보여준다

내부적으로 질문 후보는 20~40개 만들 수 있다. 하지만 사용자에게는 선택한 모드에 따라 처음 3~5개 또는 8~12개만 보여준다.

## D4. 확인 부족은 실패가 아니다

확인 부족은 검증 근거가 부족하다는 정상 상태다. 제품은 이를 실패처럼 과장하지 않는다.

## D5. 고쳐보기는 2단계다

1. 제품 설명서/결정을 고치는 loop
2. 실제 결과물/코드를 고치는 loop

## D6. 자동수정은 제한한다

한 항목당 자동수정은 최대 2회만 시도한다. 이후에는 사람 확인 필요로 넘긴다.

## D7. CLI는 사용자 제품 경험이 아니라 내부 실행 엔진이다

일반 유저에게 CLI 명령어를 요구하지 않는다. 앱에서 버튼을 누르면 뒤에서 기존 Conclave job이 실행되는 구조로 간다.

## D8. GitHub PR comment는 보조 채널이다

주요 경험은 Conclave 앱 안의 시각적 작업공간이다. GitHub PR comment에는 요약과 링크만 남긴다.
