# SaaS rollout — Problem 3 deploy guide

How to take the v0.16 central-plane SaaS endpoints from green CI to live
production on `https://conclave-ai.seunghunbae.workers.dev`.

## What's in this PR

- D1 migration `0009_saas_v1.sql` — `saas_users`, `saas_tokens`,
  `saas_device_codes`, `gh_app_installations`, `usage_meters`.
- `gh-app.ts` — webhook signature verify, JWT minting, installation token
  exchange, OAuth code exchange, `/user` lookup.
- Routes — `/webhook/github`, `/auth/device`, `/auth/token`,
  `/auth/github/callback`, `/auth/logout`, `/saas/review`, `/saas/autofix`,
  `/saas/me`.
- CLI commands — `conclave login`, `conclave logout`, `conclave whoami`.
- Token storage — `~/.conclave/auth.json` (Windows: `%USERPROFILE%\.conclave\auth.json`),
  permission-tightened on write.

## Step 1 — Apply D1 migration

```powershell
pnpm -C apps/central-plane wrangler d1 migrations apply --remote conclave-ai
```

Verifies: `saas_users` etc. created. Re-running is a no-op.

## Step 2 — Register Worker secrets

The new endpoints fail-closed (503/401) until these are set. Run from
`apps/central-plane`:

```powershell
# 1. App ID — public-ish but stored as a secret to avoid checked-in env drift
echo 3620556 | wrangler secret put GH_APP_ID

# 2. Client ID — public-ish
echo Iv23ctRM7IwJNUrALBBc | wrangler secret put GH_APP_CLIENT_ID

# 3. Client secret — generated by you on the GH App settings page →
#    "Client secrets" → "Generate a new client secret".
#    Paste the value when wrangler prompts.
wrangler secret put GH_APP_CLIENT_SECRET

# 4. Webhook secret — the random string you set during GH App registration
echo avowyuebovw09172bdsvjw6qt35f1bv | wrangler secret put GH_APP_WEBHOOK_SECRET

# 5. Private key — multi-line PEM. Pipe the whole file in:
Get-Content "C:\Users\ADMIN\.conclave\conclave-ai-code-council.2026-05-06.private-key.pem" -Raw | wrangler secret put GH_APP_PRIVATE_KEY
```

> **Note**: secret #3 (`GH_APP_CLIENT_SECRET`) hasn't been generated yet.
> The GH App settings page → "Client secrets" section has a "Generate a
> new client secret" button. Click it, copy the value (only shown once),
> paste into the wrangler prompt.

After all five are registered:

```powershell
wrangler secret list
# expect: GH_APP_ID, GH_APP_CLIENT_ID, GH_APP_CLIENT_SECRET, GH_APP_WEBHOOK_SECRET, GH_APP_PRIVATE_KEY
```

## Step 3 — Deploy Worker

```powershell
pnpm -C apps/central-plane ship
```

This runs preflight checks then `wrangler deploy`. After deploy, smoke-test:

```powershell
# Should return JSON; not 404.
curl https://conclave-ai.seunghunbae.workers.dev/healthz

# Webhook signature endpoint — should 401 unsigned, not 404.
curl -X POST https://conclave-ai.seunghunbae.workers.dev/webhook/github -d '{}'
# expected: { "error": "signature mismatch" }
```

## Rollback

```powershell
# If a deploy goes bad:
pnpm -C apps/central-plane wrangler rollback
```

D1 migration is forward-only; if `0009` causes issues, a reverse
migration would have to drop the new tables explicitly. Tables are
purely additive — older code paths don't read them — so a Worker
rollback alone is enough to unwind a bad deploy.
